# テストルール

## 基本方針

- **実装とテストは常にセット** — 新しいコードを書く際は、必ず対応するテストも同時に書くこと
- テストなしのコードはレビュー対象外とする
- バグ修正時は、まず再現テストを書いてから修正する（TDD的アプローチ）

## テストフレームワーク

- **API**: Vitest + `@cloudflare/vitest-pool-workers`（Workers環境のテスト）
- **パッケージ**: Vitest
- **モバイル**: TBD

```bash
bun run test              # 全テスト実行
bun run test -- --watch   # ウォッチモード
```

## ファイル配置

- テストファイルはソースファイルと同階層に `*.test.ts` で配置
- テストヘルパー・フィクスチャは `__tests__/` ディレクトリに配置

```
src/
  routes/
    auth.ts
    auth.test.ts          # authルートのテスト
  services/
    user.ts
    user.test.ts          # userサービスのテスト
  __tests__/
    helpers.ts            # テスト共通ヘルパー
    fixtures.ts           # テストデータ
```

## テストの種類と粒度

### ユニットテスト（必須）

- 個々の関数・サービスの動作を検証
- 外部依存（DB, R2, KV）はモックする
- 高速に実行できること

### 統合テスト（推奨）

- APIエンドポイントのリクエスト→レスポンスを検証
- `@cloudflare/vitest-pool-workers` でD1/R2/KVをローカルエミュレート
- 認証フローを含むE2Eに近いテスト

## テスト記述ルール

### 命名

- `describe` にはテスト対象のモジュール名・関数名を書く
- `it` / `test` には期待する振る舞いを日本語または英語で簡潔に書く

```typescript
describe("UserService", () => {
  describe("getById", () => {
    it("存在するユーザーを返す", async () => { ... });
    it("存在しないIDで404エラーを返す", async () => { ... });
  });
});
```

### 構造

- **Arrange / Act / Assert** パターンを守る
- 1テストにつき1アサーション（関連する複数のアサーションはOK）
- テスト間の依存・順序依存を作らない

### カバレッジの目安

| 対象 | 最低カバレッジ |
|------|---------------|
| バリデーション関数 | 正常系 + 異常系の全パターン |
| サービス層 | 主要パス + エラーパス |
| ルートハンドラ | 正常レスポンス + 各エラーステータス |
| ユーティリティ | 全パブリック関数 |

### エッジケース

以下のケースは必ずテストに含める:

- バリデーションエラー（不正入力、境界値）
- 認証・認可エラー
- リソース不在（404）
- ページネーションの境界（0件、最大件数超過）

## モック・スタブ

- D1: `@cloudflare/vitest-pool-workers` のローカルD1を使用
- R2/KV: 同上のエミュレーション、または `vi.fn()` でモック
- 外部API（OAuth, LLM）: `vi.fn()` でモック
- 時刻依存: `vi.useFakeTimers()` を使用

## テスト実行の確認

- コミット前にテストが全件パスすることを確認
- 新規テストが既存テストを壊していないことを確認
- CIでテストが通らないPRはマージしない
